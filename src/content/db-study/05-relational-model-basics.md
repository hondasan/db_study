# リレーショナルモデルとデータ独立性

物理的な実装表現に依存せず、データを論理的な集合（リレーション）として扱う「リレーショナルモデル」の定義と、アプリケーションとデータの依存性を断ち切るための「データ独立性」の概念について解説する。

---

## リレーショナルモデルの基礎

リレーショナルモデル（Relational Model）は、1970年にエドガー・F・コッド（Edgar F. Codd）によって提唱された。これは、データの物理的な保存方法（ディスク上のオフセットやメモリアドレスなど）から独立し、数学的な「関係（リレーション）」に基づいてデータを表現するデータモデルである。

### 主要な用語と定義

リレーショナルモデルを理解するための重要な基本用語を以下に整理する。

*   **リレーション / テーブル (Relation / Table)**: エンティティを表現する属性の関連性を含む、順序のないデータの集合。
*   **タプル / レコード (Tuple / Record)**: リレーション内の属性値のセットであり、テーブルの「1行分」のデータに相当する。
*   **属性 / 列 (Attribute / Column)**: タプルを構成する個々のデータ項目。名前と、許可される値の範囲（ドメイン/データ型）を持つ。
*   **プライマリキー / 主キー (Primary Key)**: テーブル内の単一のタプルを一意に識別できる属性、または複数の属性の組み合わせ。
*   **外部キー (Foreign Key)**: あるリレーションの属性が、別のリレーションのタプル（通常はプライマリキー）に対応していることを示す識別子。
*   **NULL**: あるタプルの属性値が「未定義」または「不明」であることを表す特殊な値。

---

## リレーショナルモデルを構成する3要素

リレーショナルモデルは、「構造」「整合性」「操作」の3つの要素で構成されている。

### 1. 構造 (Structure)
リレーション（テーブル）内のタプルには順序が存在しない。そのため、DBMSはデータの物理的な保存方法を自由に最適化できる。また、プライマリキーが異なれば、他の属性値が重複したレコードの存在も許容される。

### 2. 整合性 (Integrity)
データベース内のデータが満たすべき制約（整合性制約）を定義し、無効なデータが保存されるのを防ぐ。
例えば、「年齢の属性には負の値を許可しない」「メールアドレス列はNULLを禁止する」といったルールをDBMSが自動的に検証し、データの正確性を保護する。

### 3. 操作 (Manipulation)
データをどのように取得するかという低レベルの手順を指示するのではなく、「どのようなデータが欲しいか」という結果のみを指定する宣言的APIを使用する。最も効率的な実行計画の決定はDBMSに委ねられる。

<div class="note-box">
  <p><strong>設計例: 多対多の関係性をどう表現するか</strong></p>
  <p>1人のアーティストが複数のアルバムを出し、1つのアルバムに複数のアーティストが参加する「多対多（Many-to-Many）」の関係性を表現する場合、アルバムテーブルに直接アーティストIDを保存することはできない（1つのセルに複数の値を格納することになり、データの整合性が崩れるため）。</p>
  <p>この場合、<code>artists</code> と <code>albums</code> の間に、両者の関係性のみを記録する中間テーブル（交差テーブル）<code>artist_album</code> を作成し、それぞれの主キーを外部キーとして格納する設計を行う。</p>
</div>

## データ独立性 (Data Independence)

データ独立性とは、アプリケーションが動作する論理レイヤーと、データがディスク等に保存される物理レイヤーを完全に分離し、低レベルのデータ表現の変更からユーザーやアプリケーションを分離する機能である。

この独立性を実現するため、データベースの構造は以下の3つの階層（スキーマ）に分けて管理される。

1.  **物理スキーマ (Physical Schema)**: ファイル、ページなど、データがストレージにどのように物理的に保存・配置されるかの定義。
2.  **論理スキーマ (Logical Schema)**: テーブル、列、データ型、プライマリキーなど、データの論理的な構造の定義。
3.  **外部スキーマ / ビュー (External Schema / View)**: 特定のアプリケーション向けに論理スキーマをさらに抽象化し、一部を隠蔽・公開する層。

### 物理的データ独立性と論理的データ独立性

*   **物理的データ独立性**: 
    DBMSがワークロードや環境に応じて物理的なデータレイアウト（ファイルの配置やインデックス構造など）を自由に最適化・変更しても、論理スキーマは一切影響を受けない。物理スキーマが変更されても論理スキーマは影響を受けないため、アプリケーションを破壊することはない。
*   **論理的データ独立性**: 
    データベースの論理構造（テーブルの追加など）が変更されても、影響を受けるアプリケーションに対しては外部スキーマ（ビュー）を介して元の論理構造を提供し続けることができる。これにより、論理スキーマの変更が既存のクエリを破壊するリスクを排除する。

<div class="note-box">
  <p><strong>コラム: 外部スキーマによるパスワード情報の隠蔽</strong></p>
  <p>学生テーブルに含まれる「パスワード」列をすべてのアプリケーションに公開したくない場合、外部スキーマ（ビュー）を定義することで、論理層や物理層の構造を変えずに特定の列だけを隠蔽することができる。これにより、必要なデータだけを安全に外部へ開示できる。</p>
</div>

### 技術選定の理由（Why）
物理的なポインタ（ディスクのオフセットやアドレス）ではなく、論理的な「値（識別子）」を介してデータを関連付けるリレーショナルモデルとデータ独立性は、システムに高い柔軟性と保守性をもたらす。物理的なデータ構造の変更（ツリー構造からハッシュテーブルへの変更など）が発生するたびにアプリケーションのAPIやクエリを書き直す必要性をなくすため、この分離が必須とされる。
