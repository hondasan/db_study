# テーブルの結合（JOINの基本）

実際のデータベース設計では、データは整理されて複数のテーブルに分割されて保存されています。
例えば、「注文情報」と「顧客情報」は別々のテーブルに保存し、必要なときに結合して一つの表として取得します。

このテーブル同士をつなぎ合わせる操作を **JOIN（結合）** と呼びます。

## なぜテーブルを分けるのか？

例えば、1つのテーブルに「注文内容」と「顧客の住所や電話番号」をすべて保存してしまうと、同じ顧客が何度も注文した際に、その顧客の住所情報が何度も重複して保存されてしまいます。
これは無駄な容量を消費するだけでなく、住所が変わったときに全ての行を更新しなければならず、データの不整合（バグ）の元になります。

そのため、通常は以下のようにテーブルを分けます。
1.  `users` テーブル（顧客のマスター情報）
2.  `orders` テーブル（誰がいつ何を買ったかの情報。ユーザーのIDだけを記録しておく）

---

## 内部結合（INNER JOIN）

最もよく使われるのが **INNER JOIN** です。
両方のテーブルで**共通して存在するデータのみ**を結合して取得します。

```sql
SELECT orders.id, users.name, orders.amount
FROM orders
INNER JOIN users ON orders.user_id = users.id;
```

*   `FROM orders`: 結合の基準となるテーブル。
*   `INNER JOIN users`: 結合したいテーブル。
*   `ON orders.user_id = users.id`: どの列同士を紐付けるか（結合条件）を指定します。ここでは注文情報の「誰が買ったか(user_id)」とユーザー情報の「ID(id)」を紐付けています。

---

## 外部結合（LEFT JOIN）

**LEFT JOIN**（または LEFT OUTER JOIN）は、**左側（FROM句に指定した側）のテーブルのデータはすべて表示**し、右側（JOIN句に指定した側）のテーブルに一致するデータがあれば結合する、なければ `NULL`（空っぽ）を表示する結合方法です。

```sql
-- すべてのユーザーを表示し、注文があればその注文額も表示する
-- 一度も注文していないユーザーも結果に含まれます
SELECT users.name, orders.amount
FROM users
LEFT JOIN orders ON users.id = orders.user_id;
```

### LEFT JOIN が役立つ場面
「まだ一度も本を購入していないユーザーも含めて、ユーザー一覧を作りたい」
「一度もログインしていないユーザーを特定したい（ログインログがNULLのユーザー）」

RDBの本質は、このテーブル間の結合にあります。最初は少し難しく感じるかもしれませんが、実際にクエリを書いて結果を見ることで直感的に理解できるようになります。
